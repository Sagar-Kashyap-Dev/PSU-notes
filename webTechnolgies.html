    CSS Cascading specificity Calculations

    Browsers determine element styling properties by ranking overlapping style definitions using a strict structural precedence formula:

    $$\text{Specificity Weight} = (\text{Inline Styles},\, \text{IDs},\, \text{Classes/Attributes/Pseudo-classes},\, \text{Elements/Pseudo-elements})$$

    When computing styles, an explicit ID selector $(0,1,0,0)$ completely overrides any number of combined class selectors, such as ten classes matching a single target $(0,0,10,0)$.

    2. JavaScript Engine Runtime, DOM Tree & Events

    The Document Object Model (DOM) Hierarchy

    The browser parses HTML source code into a live, interactive tree structure called the Document Object Model (DOM). Each node in the tree represents an element, attribute, or text segment. JavaScript interacts with this object tree to dynamically alter page content, modify styles, and handle user interactions.

    The Asynchronous JavaScript Event Loop Architecture

    The engine manages single-threaded execution using a structured concurrency model:

    • Call Stack: Tracks active function execution contexts sequentially using standard LIFO behaviors.
    • Web APIs Framework: Handles background threads for long-running operations like timers (`setTimeout`) and asynchronous network calls (`fetch`).
    • Callback/Task Queue: Stores deferred macro-tasks until the main execution stack clears.
    • Microtask Queue: Prioritizes immediate operations (such as resolving `Promise` handlers) before processing items from the standard Task Queue.
    Event Propagation Mechanics

    When an interaction occurs, the event moves through a three-phase execution cycle across the target tree:

    1. Capturing Phase: The event propagates down from the top root window to the nested target node.
    2. Target Phase: Event listeners attached directly to the initiating element trigger execution.
    3. Bubbling Phase: The event travels back up the tree from the target node to the root. This upward flow can be halted at any point by calling `event.stopPropagation()`.

    3. Network Serialization (XML/JSON) & HTTP Methods

    Data Serialization Engine Formats
    • XML (Extensible Markup Language): A strict text serialization standard. It supports advanced user-defined schemas, enforces type safety via **DTDs** or **XML Schemas (XSD)**, and requires formal parsers to traverse its verbose tag structures.
    • JSON (JavaScript Object Notation): A lightweight, human-readable data serialization format based on key-value pairs. It natively supports primitive data types (Strings, Numbers, Booleans, Arrays, and Null), making it the preferred format for modern high-performance web APIs.

    4. Stateless Protocols, State Mechanics & Security

    HTTP: Stateless Protocol Layer
    Cookies: Client-Side Storage Strings
    Sessions: Server-Side In-Memory State Mapping
    State Enforcement (Cookies vs. Sessions)
    • Because HTTP is stateless, servers use persistent data stores to track users across requests.
    • HTTP Cookies: Small text payloads sent from the server and saved in the client's browser. They are automatically appended to the `Cookie` header of every subsequent request to that domain. Essential security flags include `HttpOnly` (blocks access via JavaScript to mitigate Cross-Site Scripting) and `Secure` (restricts transmission to encrypted HTTPS connections).
    • Server-Side Sessions: Storage maps maintained on the server. The client receives a randomized tracking token called a `Session ID` via a cookie. When a new request arrives, the server checks this ID against its memory or database to load the user's active session state.

    5. Web Services Architecture: SOAP vs. REST

    RESTful API Design Constraints

    Architectures must adhere to six core architectural principles to be considered truly RESTful:

    1. Statelessness: Every request must contain all the contextual information needed for processing; the server cannot store any persistent session state about the client.
    2. Client-Server Separation: Decouples client interface concerns from server data storage paths, letting each side scale independently.
    3. Cacheability: Server responses must explicitly label data as cacheable or non-cacheable to improve network efficiency.
    4. Uniform Interface: Simplifies interaction models by standardizing resource URI formats and leveraging native HTTP verbs.
    5. Layered System: The client cannot tell whether it is connected directly to the end server or to an intermediary layer like a proxy or load balancer.
    6. Code on Demand (Optional): Allows the server to temporarily extend client functionality by sending executable scripts.
