1. Arrays & Sparse Matrix

Type: Linear DS
Memory: Contiguous Block
Access: O(1)
Traversal: O(n)
Binary Search: O(log n)
Insertion/Deletion (Middle): O(n)
Cache: Highly Friendly
Row-Major: C/C++
Column-Major: Fortran
Definition & Structural Properties
  • Array Definition: A finite, ordered collection of homogeneous elements mapped directly across a single block of contiguous memory locations.
  • Sparse Matrix Definition: A matrix of dimension $M \times N$ where the count of zero elements is significantly greater than the count of non-zero elements ($\text{Zeros} \gg \text{Non-Zeros}$).
  • Static Array: Fixed-size structures allocated at compile-time on the stack; size boundaries cannot be mutated during runtime execution.
  • Dynamic Array: Resizable array structures allocated on the runtime heap. When capacity is reached, it dynamically provisions a new block (typically using a doubling growth strategy factor of $2\times$), copying elements in $\mathcal{O}(N)$ time, which averages out to an amortized $\mathcal{O}(1)$ insertion cost.
Memory Layout Mapping & Address Formulas
  • Physical Contiguity: Memory addresses follow a strict sequential layout. Knowing the base address and the element size is all that is needed to calculate the exact address of any item.
  • Row-Major Order (RMO): Elements are stored sequentially, row by row. Used by default in C and C++.
    Formula for $A[i][j]$ in a 2D array of size $M \times N$ (0-indexed): $$\text{Address}(A[i][j]) = \text{Base} + (i \times N + j) \times \text{size}$$
  • Column-Major Order (CMO): Elements are stored sequentially, column by column. Used by default in Fortran.
    Formula for $A[i][j]$ in a 2D array of size $M \times N$ (0-indexed): $$\text{Address}(A[i][j]) = \text{Base} + (j \times M + i) \times \text{size}$$
  • General Upper/Lower Bound Formula (1-Indexed or Arbitrary Range $L_1 \dots U_1, L_2 \dots U_2$):
    Let $\text{Rows} = (U_1 - L_1 + 1)$ and $\text{Cols} = (U_2 - L_2 + 1)$. $$\text{RMO Address}(A[i][j]) = \text{Base} + \left[(i - L_1) \times \text{Cols} + (j - L_2)\right] \times \text{size}$$ $$\text{CMO Address}(A[i][j]) = \text{Base} + \left[(j - L_2) \times \text{Rows} + (i - L_1)\right] \times \text{size}$$
  • Sparse Matrix Triplet Storage: Represented as an $K \times 3$ matrix array where $K$ is the number of non-zero elements. Row 0 stores matrix metadata: `[Total Rows, Total Columns, Total Non-Zero Elements]`. Each subsequent row explicitly stores a data element's coordinates: `[Row_Index, Column_Index, Value]`.

Engineering Trade-offs & Comparisons
  • Advantages: Instantaneous $O(1)$ index access; excellent **cache locality** and high hardware cache hit ratios due to sequential memory locations.
  • Disadvantages: Fixed sizes can result in wasted space or out-of-memory errors; inserting and deleting elements introduces performance overhead because items must be shifted in memory to close or create gaps.
  • Comparison with Linked Lists: Arrays provide fast $O(1)$ random lookups compared to the slower $O(N)$ sequential lookups in linked lists. However, linked lists offer flexible, true $O(1)$ local insertions and deletions without needing to move adjacent data.
Core Structural Applications
  • Primary building block for Hash Tables, Binary Heaps, and matrices.
  • Used to implement lookup tables, CPU registers, and string operations.
Common Exam Traps & Pitfalls:
  • Index Offset Shifting: Always check whether the problem states the array is 0-indexed or 1-indexed. Forgetting to subtract 1 from indices in 1-indexed problems will lead to incorrect address calculations.
  • Incorrect Dimension Multipliers: When calculating addresses in Row-Major Order, always multiply the row offset by the total number of **columns** ($N$), not the total number of rows.
  • Dynamic Expansion Bounds: Remember that a dynamic array's worst-case insertion time is $\mathcal{O}(N)$ when resizing occurs, but its **amortized** insertion time remains a constant $\mathcal{O}(1)$.

2. Linked Lists (Singly, Doubly, Circular)

Type: Dynamic Linear DS
Memory: Non-Contiguous (Heap)
Random Access: O(n)
Insert at Head: O(1)
Pointer Overhead: High
Cache Hit: Poor Locality
Definition & Structural Layouts
  • Linked List Definition: A dynamic linear data structure where elements are stored in separate node objects. Their logical order is maintained by explicit pointer references rather than physical proximity in memory.
  • Singly Linked List Node: Composed of an element data field and a single address pointer (`next`) targeting the following node. The last node's pointer targets `NULL`.
  • Doubly Linked List Node: Composed of an element data field and two tracking pointers: `next` (targeting the subsequent node) and `prev` (targeting the preceding node). This allows full bidirectional traversal.
  • Circular Linked List Node: A variation where the last node's `next` pointer points back to the `head` node instead of terminating at `NULL`, closing the structure into a continuous loop.
Memory Allocation & Cache Behavior
  • Heap Allocation: Individual node allocations are processed dynamically using heap managers (like `malloc` or `new`). Nodes can be scattered across non-contiguous physical addresses.
  • Cache Line Misses: Because nodes are stored in non-contiguous locations, traversing a linked list frequently causes cache misses, making it less performant than an array for raw sequential processing.
  • Memory Overhead: Requires extra memory to store pointers alongside data fields. On standard 64-bit architectures, each pointer introduces an 8-byte tracking overhead.

Engineering Trade-offs & Comparisons
  • Advantages: Dynamic resizing at runtime; elements can be inserted or deleted locally in $\mathcal{O}(1)$ time without needing to shift data in memory.
  • Disadvantages: Does not support random access; searching requires sequential traversals; added pointer storage overhead.
  • Singly vs. Doubly Lists: Doubly linked lists allow you to delete a given node in true $\mathcal{O}(1)$ time because the node already contains a link to its predecessor. Singly linked lists require an $\mathcal{O}(N)$ search to find the preceding node before updating references.
Common Exam Traps & Pitfalls:
  • Orphaned Pointer Subtrees: Always update link pointers in the correct order. For example, when inserting node `X` after node `P` in a singly linked list, you must set `X->next = P->next` **before** updating `P->next = X`. Reversing this order will orphan the rest of the list.
  • Circular Traversal Infinity: When looping through a circular list, terminating loops with conditions like `while(curr != NULL)` will cause an infinite loop. Instead, terminate the loop when the tracker returns to the starting node: `while(curr != head)`.
  • Tail Deletion in Singly Lists: Even if a singly linked list maintains an explicit `tail` pointer, deleting the last node still takes $\mathcal{O}(N)$ time. This is because you must traverse the list from the head to find the second-to-last node and update its `next` pointer to `NULL`.

3. Stack & Expression Mechanics

Type: Restricted Linear DS
Protocol: LIFO (Last In First Out)
Core Operations: Push / Pop / Peek
Operation Complexity: O(1) Strict
Infix to Postfix: 1 Left-to-Right Scan
Definition & Structural Constraints
  • Stack Definition: A restricted linear data structure that follows the **Last-In, First-Out (LIFO)** access protocol. Elements can only be added or removed from a single end called the **Top**.
  • Array Implementation: Uses a fixed-size array along with a scalar tracking variable, `top`. An index check for `top == Capacity - 1` signals a **Stack Overflow** condition, while `top == -1` signals a **Stack Underflow** condition.
  • Linked List Implementation: Uses a singly linked list where insertions and deletions happen exclusively at the `head` node. This approach avoids stack overflows since memory is allocated dynamically on the heap.
Core Operations & Variations
  • Push: Adds an element to the top of the stack. Takes $\mathcal{O}(1)$ time.
  • Pop: Removes and returns the top element from the stack. Takes $\mathcal{O}(1)$ time.
  • Peek/Top: Returns the top element without removing it. Takes $\mathcal{O}(1)$ time.
Expression Parsing Frameworks (Infix, Postfix, Prefix)

Comprehensive Algorithmic Conversions
  • Infix to Postfix Algorithm (Shunting-yard):
    1. Scan the infix string from **Left to Right**.
    2. If the scanned character is an **operand**, append it immediately to the output string.
    3. If the character is an **open parenthesis `(`**, push it directly onto the operator stack.
    4. If the character is a **closing parenthesis `)`**, pop elements off the stack and append them to the output string until an open parenthesis `(` is encountered. Pop and discard the open parenthesis.
    5. If an **operator** is encountered, pop operators off the stack and append them to the output string if they have **greater than or equal precedence** to the current operator. Then, push the current operator onto the stack.
  • Operator Associativity Exception: When dealing with the right-associative exponentiation operator (`^`), a incoming `^` does **not** pop an existing `^` off the stack, because right-associative operators evaluate from right to left.
Core Structural Applications
  • Managing function calls and activation records on the runtime stack frame.
  • Implementing undo/redo histories, browser navigation history, and matching brackets/parentheses in compilers.
Common Exam Traps & Pitfalls:
  • Operand Order in Postfix Evaluation: When evaluating a postfix expression, popping an operator means the first item popped is the **right operand** and the second item popped is the **left operand**. For example, for the sequence `A B -`, popping `B` then `A` must be evaluated as `A - B`, not `B - A`. This distinction is crucial for non-commutative operators like subtraction and division.
  • Permutation Generation Limits: A common exam question asks if a given sequence can be generated by a stack. For an input sequence $1, 2, 3 \dots N$, a output permutation $p_1, p_2 \dots p_n$ is **impossible** if there exist indices where $i < j < k$ such that $p_k < p_i < p_j$.

4. Queues (Circular, Priority Queue, Deque)

Protocol: FIFO (First In First Out)
Pointers: Front (Delete) / Rear (Insert)
Circular Condition: (Rear + 1) % Cap == Front
Priority Queue: O(log n) Heap Bounds
Deque: Double-Ended Access
Definition & Structural Implementations
  • Queue Definition: A restricted linear data structure that follows the **First-In, First-Out (FIFO)** operational protocol. Elements are added at one end, called the **Rear** (Enqueue), and removed from the opposite end, called the **Front** (Dequeue).
  • Linear Array Queue Limitations: In a basic linear array layout, deleting items advances the `front` pointer forward. This space cannot be reused without shifting elements, which can cause false overflow errors where space is available but the `rear` pointer has reached the end of the array.
  • Circular Queue Architecture: Solves linear array limitations by wrapping indices back to the beginning of the array using modulo arithmetic:
    Rear = (Rear + 1) % Capacity
    Front = (Front + 1) % Capacity
Advanced Structural Variations
  • Deque (Double-Ended Queue): A flexible queue variant that supports insertions and deletions at both the Front and Rear boundaries.
    • Input-Restricted Deque: Insertions are restricted to the rear, but deletions can be performed at both ends.
    • Output-Restricted Deque: Deletions are restricted to the front, but insertions can be performed at both ends.
  • Priority Queue: Elements are assigned priority values, and items with higher priorities are served before lower priority items. Arrival order is ignored if priorities differ. This structure is most efficiently implemented using a **Binary Heap**.

Core Structural Applications
  • Asynchronous data buffering for IO pipelines (such as print spools or disk writing buffers).
  • Operating system CPU task scheduling queues (like Round-Robin queues) and network packet buffering.
Common Exam Traps & Pitfalls:
  • Circular Full vs Empty Confusion: In a standard circular queue implementation, the conditions for an empty queue and a full queue both result in `Front == Rear`. To fix this, look at how the problem resolves the conflict: either it leaves one slot empty to flag a full queue using `(Rear + 1) % Capacity == Front`, or it tracks status using an explicit counter variable.
  • Stack-to-Queue Transformation Costs: Implementing a queue using two stacks requires transferring elements between them. While individual operations can take $\mathcal{O}(N)$ time in the worst case during a transfer, the **amortized** time complexity for each operation over a series of actions remains a highly efficient $\mathcal{O}(1)$.

5. Trees (Binary Tree, BST, AVL, Heap, Threaded)

Type: Hierarchical Non-Linear DS
BST Inorder: Always Sorted
AVL Balance Factor: {-1, 0, +1} Strict
Max-Heap Root: Maximum Element
Threaded Tree: No Null Pointers
Comprehensive Tree Classifications
  • General Binary Tree: A hierarchical structure where each node can have at most two children, designated as the left child and the right child.
  • Binary Search Tree (BST): Enforces an ordering property where for any given node, all keys in its left subtree must be strictly less than the node's key, and all keys in its right subtree must be strictly greater than the node's key.
  • AVL Tree: A self-balancing Binary Search Tree where the height difference between the left and right subtrees (the **Balance Factor**) of any node cannot exceed $\pm 1$. $$\text{Balance Factor} = \text{Height}(\text{Left Subtree}) - \text{Height}(\text{Right Subtree}) \in \{-1, 0, 1\}$$
  • Binary Heap: A complete binary tree that satisfies the heap property:
    • *Max-Heap:* Every parent node's key is greater than or equal to the keys of its children. The absolute maximum element is always at the root.
    • *Min-Heap:* Every parent node's key is less than or equal to the keys of its children. The absolute minimum element is always at the root.
  • Threaded Binary Tree: Replaces unused `NULL` pointers with structural threads. In an inorder threaded tree, a left `NULL` pointer is replaced with a thread pointing to the node's **inorder predecessor**, and a right `NULL` pointer is replaced with a thread pointing to its **inorder successor**. This allows tree traversals without using a stack or recursion.
Memory Allocation Frameworks
  • Linked Representation: Each node is allocated dynamically as an object containing data along with `left_child` and `right_child` address pointers.
  • Sequential Array Representation (Heaps): Perfect for Complete Binary Trees. Elements are mapped directly to flat array positions without using pointers:
    • Parent of node at index $i$ is located at index: $\lfloor \frac{i-1}{2} \rfloor$
    • Left child of node at index $i$ is located at index: $2i + 1$
    • Right child of node at index $i$ is located at index: $2i + 2$

Essential Mathematical Axioms
  • Handshaking Lemma variant for Binary Trees: If a binary tree has $n_0$ leaf nodes and $n_2$ nodes with exactly two children, then: $$n_0 = n_2 + 1$$
  • Strict Catalan Combinations: The total number of unique structural binary tree shapes that can be formed using $N$ distinct keys is given by the $N^{\text{th}}$ Catalan number: $$T(N) = \frac{1}{N+1} \binom{2N}{N}$$
  • Heap Construction Limits: Building a binary heap from an arbitrary array of $N$ elements using the bottom-up `Build-Heap` method takes $\mathcal{O}(N)$ time, which is more efficient than inserting elements one by one ($\mathcal{O}(N \log N)$).
Common Exam Traps & Pitfalls:
  • Unique Reconstruction Constraints: You cannot reconstruct a unique binary tree using only its **Preorder** and **Postorder** traversals unless the tree is guaranteed to be a Full Binary Tree (where every node has either 0 or 2 children). To uniquely reconstruct any standard binary tree, you must have the **Inorder** traversal along with either the Preorder or Postorder traversal.
  • BST Deletion Successor Swap: When deleting a node with two children from a BST, you swap its value with either its inorder predecessor or its inorder successor. Remember that this replacement node will **always** have at most one child, making its subsequent removal from its original position straightforward.
  • AVL Rotation Target Identification: When an insertion unbalances an AVL tree, look for the **deepest** node whose balance factor becomes invalid ($\pm 2$). Rotations must be applied to this specific node to restore balance across the tree.

6. Graphs (Representation, BFS, DFS, Connected Components, Articulation Points)

Type: Non-Linear Network DS
BFS Structure: Queue Queue
DFS Structure: Stack / Recursion
Traversal Complexity: O(V + E)
Matrix Space: O(V²)
List Space: O(V + E)
Definition & Structural Layouts
  • Graph Definition: A non-linear data structure consisting of a finite set of Vertices ($V$) along with a set of Edges ($E$) that connect pairs of vertices. Directed graphs have edges with specific directions, while undirected graphs have bidirectional connections.
  • Adjacency Matrix Representation: A 2D array of size $V \times V$ where `Matrix[i][j] = 1` indicates an edge from vertex $i$ to vertex $j$.
    • *Space Complexity Overhead:* $\mathcal{O}(V^2)$, regardless of the number of edges. Excellent for **dense graphs** ($E \approx V^2$).
    • *Edge Verification:* Checking if an edge exists between two vertices takes constant $\mathcal{O}(1)$ time.
  • Adjacency List Representation: An array of size $V$ where each entry points to a linked list containing all neighboring vertices.
    • *Space Complexity Overhead:* $\mathcal{O}(V + E)$ for directed graphs, and $\mathcal{O}(V + 2E)$ for undirected graphs. Ideal for **sparse graphs** ($E \ll V^2$).
    • *Edge Verification:* Checking if an edge exists requires traversing a vertex's neighbor list, taking $\mathcal{O}(\text{degree}(V))$ time.
Core Traversal Algorithmic Pipelines
  • Breadth-First Search (BFS): Explores the graph level by level, radiating outward from a starting vertex.
    • *Tracking Structure:* Uses an explicit **Queue** along with an array to track visited vertices.
    • *Application:* Finds the shortest path between two vertices in an unweighted graph.
    • *Time Complexity:* $\mathcal{O}(V + E)$ when using adjacency lists, and $\mathcal{O}(V^2)$ when using adjacency matrices.
  • Depth-First Search (DFS): Explores as deep as possible along each branch before backtracking to explore alternative paths.
    • *Tracking Structure:* Uses the system call stack or an explicit **Stack** structure along with a visited array.
    • *Edge Classifications:* Categorizes graph edges into Tree Edges, Back Edges, Forward Edges, and Cross Edges.
    • *Time Complexity:* $\mathcal{O}(V + E)$ when using adjacency lists.
Advanced Property Algorithms
  • Connected Components: Maximal subgraphs in an undirected graph where a path exists between any pair of vertices. The total count of connected components can be found by tracking how many times a full BFS or DFS traversal must be restarted to visit every vertex.
  • Articulation Point Definition: A vertex in a connected undirected graph whose removal (along with its connected edges) disconnects the graph, increasing the total number of connected components.
  • Tarjan's Articulation Point Discovery: Uses a single DFS pass to track two values for each vertex: `discovery_time[u]` (when the vertex was first visited) and `low_link[u]` (the lowest discovery time reachable from $u$ using at most one back edge). A vertex $u$ is an articulation point if:
    1. It is the root of the DFS tree and has **two or more independent children**.
    2. It is not the root of the DFS tree and has a child $v$ such that: `low_link[v] >= discovery_time[u]`.
Common Exam Traps & Pitfalls:
  • Adjacency Matrix Symmetry: The adjacency matrix of an undirected graph is always symmetric across its main diagonal. For a graph with no self-loops, the diagonal elements are all 0, and the total number of edges in the graph equals the number of 1s in the matrix divided by 2.
  • Cycle Detection Criteria: In an undirected graph, finding a **Back Edge** during a DFS traversal indicates the presence of a cycle. However, in a directed graph, finding a cross edge or forward edge does not guarantee a cycle; a cycle is confirmed if and only if a back edge targets a vertex that is currently active on the DFS recursion stack.
  • DFS Tree Root Articulation: When evaluating whether the root of a DFS tree is an articulation point, do not simply count its total connections (degree) in the graph. The root is an articulation point if and only if it has **two or more distinct child branches in the generated DFS tree layout**, meaning the traversal had to backtrack completely to the root before it could explore the second child.

7. Hashing Techniques

Average Search: O(1) Strict
Worst-Case Search: O(n) Degenerate
Chaining: Open List Array Links
Linear Probing: Primary Clustering Trap
Double Hashing: Step Size = h2(k)
Load Factor (λ): Elements / Capacity
Definition & Core Mechanics
  • Hashing Definition: A technique that maps variable-sized keys to fixed-size indices in an array (a Hash Table) using a mathematical **Hash Function**. This provides highly efficient, near-$O(1)$ average-case performance for lookups, insertions, and deletions.
  • Load Factor ($\lambda$): The ratio of the number of stored elements ($N$) to the total capacity of the hash table ($M$): $$\lambda = \frac{N}{M}$$
  • Hash Functions: Good hash functions should uniformally distribute keys across the table to minimize collisions. Common methods include:
    • *Division Method:* `h(k) = k % M`, where choosing a prime number for $M$ helps reduce collisions.
    • *Multiplication Method:* `h(k) = floor(M * ( (k * A) % 1 ))`, where $A$ is a constant fractional value.
Collision Resolution Frameworks
  • Chaining (Separate Chaining / Open Enlistment): The hash table is structured as an array of pointers, where each slot links to an independent linked list containing all keys that map to the same index.
    The average time to perform a successful search is bounded by the load factor: $\mathcal{O}(1 + \lambda)$.
  • Open Addressing (Closed Hashing): All elements are stored directly within the hash table array itself. If a target slot is already occupied, the algorithm searches for an alternative open slot based on a probing strategy:
    • Linear Probing: Checks sequential slots one by one: `h(k, i) = (h'(k) + i) % M`. This simple approach can cause keys to bunch together, creating large blocks of occupied slots known as **Primary Clustering**, which degrades search efficiency.
    • Quadratic Probing: Checks alternative slots using a quadratic offset: `h(k, i) = (h'(k) + c1*i + c2*i^2) % M`. This prevents primary clustering but can lead to **Secondary Clustering**, where keys with the same initial hash value follow the exact same probe sequence.
    • Double Hashing: Uses a second independent hash function to calculate a custom step size for each key: `h(k, i) = (h1(k) + i * h2(k)) % M`. This effectively eliminates both primary and secondary clustering. The value returned by $h_2(k)$ must be relatively prime to the table size $M$ to ensure the probe sequence can check every slot in the table.
  • Rehashing: When the table's load factor exceeds a set limit (typically $\lambda > 0.75$), performance drops. The table size is doubled to the next nearest prime number, and all existing keys are re-mapped using an updated hash function.

Common Exam Traps & Pitfalls:
  • The Open Addressing Deletion Error: In an open addressing hash table, you cannot simply delete an element by clearing its slot to `NULL`. If you do, subsequent search operations for other keys that collided past that slot will stop early and incorrectly report that the key does not exist. To prevent this, slots must be marked with a special placeholder flag (such as `Deleted` or `Dummy Node`) to tell the search algorithm to keep probing.
  • Double Hashing Step Size Rule: For double hashing to work correctly, the second hash function $h_2(k)$ must never return a value of `0`. A step size of 0 would trap the algorithm in an infinite loop on the very first collision.
  • Unsorted Hash Constraint: Always remember that while hashing offers fast average-case lookups, it does not maintain elements in any sorted order. Operations like finding the minimum element, maximum element, or printing keys in sorted order require a full scan of the table, taking $\mathcal{O}(M + N)$ time.
