1. OS Basics & Services

Batch: Non-Interactive Jobs
Time-Sharing: CPU Multitasking Slots
Real-Time: Strict Deterministic Bounds
Dual-Mode: User (1) vs. Kernel (0)
Taxonomy of Operating Systems
  • Batch OS: Automates execution by grouping similar jobs without interactive user control. Minimizes operator overhead but suffers from high turnaround latency.
  • Time-Sharing OS: Utilizes clock-driven context switches to rotate the CPU among active users, providing an interactive environment.
  • Real-Time OS (RTOS): Guarantees completion within strict maximum timing limits. Split into *Hard RTOS* (missing a deadline equals critical failure) and *Soft RTOS* (deadlines are preferred but not catastrophic if missed).
System Execution Subsystems & Services

The kernel provides programmatic access via **System Calls** (e.g., `fork()`, `exec()`, `wait()`). Hardware isolation is enforced by the **Dual-Mode Execution Model** using a hardware mode bit:

  • User Mode (Bit 1): Restricts access to sensitive hardware registers and memory areas. Attempting direct access triggers a hardware trap.
  • Kernel Mode / Supervisor Mode (Bit 0): Unrestricted access to physical devices, memory registers, and raw execution paths.

2. Process Architecture & Threading Models

PCB: State Context Control Block
User Threads: Fast Non-Kernel Contexts
Kernel Threads: Scheduler Aware Threads
Context Switch: State Save & Restore
The Process Control Block (PCB) Structure

A data structure in protected kernel memory that tracks a process. It contains: the Process ID (PID), Process State, Program Counter (PC), CPU Registers, Memory Allocation Limits, and Open File Descriptors.

Process State Transitions Diagram

Processes cycle through a finite state machine managed by schedulers:

  • New $\to$ Ready: Long-Term Scheduler regulates the degree of multiprogramming.
  • Ready $\to$ Running: Short-Term Scheduler dispatches the process using chosen allocation rules.
  • Running $\to$ Ready: Interrupted due to clock slice expiration or higher priority preemption.
  • Running $\to$ Waiting: Blocks on an asynchronous I/O request or synchronization condition.

3. CPU Scheduling Mechanics

FCFS: Convoy Effect Risk
SJF: Provably Optimal Wait Profile
SRTF: Preemptive Mini-Remaining
Round Robin: Quantum-Driven Multitasking
Performance Metrics Calculations
  • $$\text{Turnaround Time (TAT)} = \text{Completion Time} - \text{Arrival Time}$$
  • $$\text{Waiting Time (WT)} = \text{Turnaround Time} - \text{Burst Time}$$
  • $$\text{Response Time (RT)} = \text{First Allocation Time} - \text{Arrival Time}$$
Scheduling Algorithms Engine Profile
  • First-Come, First-Served (FCFS): Non-preemptive. Vulnerable to the **Convoy Effect**, where short processes wait a long time behind a single compute-bound, heavy process.
  • Shortest Job First (SJF): Non-preemptive. Selects the process with the shortest CPU burst. **Provably optimal** for minimizing average waiting times.
  • Shortest Remaining Time First (SRTF): Preemptive variant of SJF. Preempts the running process if a new arriving process has a shorter remaining burst time. Suffer from process **starvation** for long-running tasks.
  • Round Robin (RR): Preemptive. Assigns the CPU to tasks for a fixed time slice (Quantum $q$). Performance scales with the quantum size: if $q \to \infty$, it degrades into FCFS; if $q \to 0$, context-switch overhead dominates the system.

4. Process Synchronization Primitives

Mutex: Binary Locking Primitives
Semaphore: Integer State Control
Counting Sem: Bounded Resource Control
Monitor: Object-Level Isolation Encapsulation
The Bounded Critical Section Criteria

Any solution protecting a critical section from a race condition must satisfy three strict requirements:

  1. Mutual Exclusion: If process $P_i$ is executing inside its critical section, no other processes can enter their critical sections for that resource.
  2. Progress: If no process is in its critical section, only processes wishing to enter can participate in deciding which process enters next, and this choice cannot be postponed indefinitely.
  3. Bounded Waiting: There must be a limit on the number of times other processes can enter their critical sections after a process has requested entry, before that request is granted. Prevents process starvation.
Semaphore Operations Framework

An integer variable $S$ accessed exclusively via two atomic, non-divisible operations:

  • Wait ($P(S)$): Decrements the resource counter: $$P(S): \text{ while } S \le 0 \text{ do skip}; \quad S = S - 1;$$
  • Signal ($V(S)$): Increments the resource counter: $$V(S): S = S + 1;$$
  • *Counting Semaphore Range:* S can take any integer value. A negative value indicates the number of processes currently blocked, waiting for that semaphore.
Critical Primitives Contrast:
  • Mutex vs. Binary Semaphore Ownership: A **Mutex is an ownership-driven mechanism**. Only the specific execution thread that acquired the mutex lock can release it. A **Semaphore is a signaling mechanism**. Any active thread can signal ($V(S)$) a binary semaphore to unblock a waiting process, regardless of which thread originally called wait ($P(S)$).

5. Deadlock Handling & The Banker's Framework

Mutual Exclusion: Exclusive Allocation
Hold & Wait: Retain and Request
No Preemption: Voluntary Release Only
Circular Wait: Cyclic Dependency Chain
The Four Coffman Conditions

Deadlocks can occur if and only if all four conditions hold simultaneously:

  • Mutual Exclusion: At least one resource must be held in a non-shareable mode.
  • Hold and Wait: A process must currently hold at least one resource while waiting to acquire additional resources held by other processes.
  • No Preemption: Resources cannot be forcibly taken from a process; they must be released voluntarily after the process finishes its task.
  • Circular Wait: A closed chain of processes exists, where each process holds one or more resources needed by the next process in the chain.
Deadlock Mitigation Systems
  • Prevention: Eliminates deadlocks by design by breaking at least one of the four Coffman conditions (e.g., enforcing a strict global ordering on resource acquisition to eliminate Circular Wait).
  • Avoidance (Banker's Algorithm): Dynamic monitoring strategy. The OS tracks resource allocation states and only grants requests if the resulting system state remains **Safe** (meaning a sequence exists ensuring all processes can eventually finish).
    Matrix Validation Condition Equation: $$\text{Need}[i][j] = \text{Max}[i][j] - \text{Allocation}[i][j]$$ A request is granted only if: $\text{Request} \le \text{Available}$ AND $\text{Request} \le \text{Need}$.

6. Memory Management Foundations

Paging: Fixed Physical Allocation
Segmentation: Variable Logical Views
Internal Frag: Unused Allocated Block Space
External Frag: Unallocated Contiguous Gaps
Paging Mechanics Framework

Divides physical memory into fixed-size blocks called **Frames** and logical memory into blocks of the same size called **Pages**. Eliminates external fragmentation but can cause **Internal Fragmentation** (wasted space in the last allocated page).

Multi-Level Paging Address Translation Architecture

Used to avoid storing large, contiguous page tables in main memory. For a linear logical address space mapped through a multi-level index:

$$\text{Logical Address Structure: } \langle p_1, p_2, \dots, p_k, d \rangle$$

Where $d$ represents the page offset, bounded directly by the physical page size ($2^d = \text{Page Size}$).

  • Page Table Entry (PTE) Components: Frame Number, Valid/Invalid bit, Dirty bit, Reference bit, and Protection bits.
Segmentation Principles

Maps logical addresses to variable-sized blocks based on user-defined structural modules (e.g., code, stack, heap). Prone to **External Fragmentation**, where total free space is sufficient to satisfy a allocation request, but the space is not contiguous.

7. Virtual Memory & Page Replacement Engines

FIFO: Queue Order Eviction
Belady's: More Frames, More Faults Anomaly
LRU: Past Reference Eviction
Optimal: Future Reference Horizon
Page Eviction Algorithms Matrix
  • First-In, First-Out (FIFO): Evicts the oldest page brought into memory. Simple to manage but vulnerable to **Belady's Anomaly**, where allocating more physical page frames results in an *increase* in the number of page faults for certain reference strings.
  • Least Recently Used (LRU): Evicts the page that has not been accessed for the longest period. Relies on the principle of temporal locality. Does not suffer from Belady's Anomaly.
  • Optimal Algorithm (OPT): Evicts the page that will not be used for the longest period in the future. Serves as a theoretical baseline for performance evaluations, as it requires perfect knowledge of future references.
Thrashing Dynamics

Occurs when a process spends more time swapping pages in and out of disk than executing instructions. This happens when a process's allocated page frames drop below the minimum required to support its active **Working Set Model** ($\sum \text{Working Sets} > \text{Total Memory}$).

8. File Systems & Disk Scheduling Core

File Allocation Strategies
  • Contiguous Allocation: Fast sequential access, but prone to external fragmentation.
  • Linked Allocation: Flexible size growth with zero external fragmentation, but slow random access and vulnerable to pointer corruption.
  • Indexed Allocation (e.g., Unix Inode): Uses multi-level index blocks (direct, single indirect, double indirect pointers) to support large files and fast random access.
