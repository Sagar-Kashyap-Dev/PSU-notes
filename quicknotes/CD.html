1. Compiler Basics vs. Interpreter

Core Operations
  • Compiler: Translates a High-Level Language (HLL) source code layout in its entirety down to target Machine Language / Assembly. Execution runs through two global operational blocks: the Analysis Phase (front-end) and the Synthesis Phase (back-end).
Operating Parameter Compiler Engine Interpreter Engine
Translation Unit Processes and translates the entire program file at once. Translates and executes code step-by-step, line-by-line.
Execution Speed Much faster runtime execution (compiled machine binary runs directly). Slower execution due to simultaneous on-the-fly parsing overhead.
Error Diagnostics Reports all detected syntax/semantic errors together after compilation completes. Halts execution immediately upon hitting an error on that specific active line.

2. The 6 Phases of a Compiler

Sequential Execution Flow
  • 1. Lexical Analysis (Scanner): Scans raw source character streams and maps them to clean Tokens.
  • 2. Syntax Analysis (Parser): Verifies grammar rules and constructs a structured Parse/Syntax Tree.
  • 3. Semantic Analysis: Validates semantic consistency and checks rule meanings (e.g., matching data types).
  • 4. Intermediate Code Generation (ICG): Generates a machine-independent stream (e.g., Three-Address Code).
  • 5. Code Optimization: Enhances code efficiency by reducing execution time, memory usage, or instruction size.
  • 6. Code Generation: Maps intermediate code down into live machine code instructions utilizing target register allocations.
Global Supporting Subsystems
  • Symbol Table: Central repository tracking variable metadata, data types, scope access metrics, and layout memory offsets.
  • Error Handler: Coordinates warning traps and structural recovery across every single phase boundary.
High-Frequency MCQ Trap Matrix
Lexical Analysis = Token Breakdown  |  Syntax Analysis = Structural Grammar  |  Semantic Analysis = Meaning Validation

3. Lexical Analysis Deep Dive (Scanner)

Core Operations
  • Transforms input character source streams into discrete, atomic lexical tokens.
  • Strips out unnecessary source formatting like **white spaces, inline tabs, and developer comments**.
  • Token Classifications: Keywords (int, while), Identifiers (variable names), Operators (+, &&), Constants (45, "hello").
Theoretical Foundations & Tooling
  • Tokens are specified formally using Regular Expressions.
  • The underlying mathematical engine that implements a lexer scanner is a Deterministic Finite Automaton (DFA).

4. Syntax Analysis Foundation (Parser)

Core Operations
  • Evaluates token streams against language grammar rules to ensure correct structural syntax.
  • Outputs a highly structured hierarchical **Parse Tree** or **Abstract Syntax Tree (AST)**.
Parser Architectures
  • Top-Down Parsers: Builds the hierarchy starting from the root start symbol down to the terminal tokens (e.g., LL(1)).
  • Bottom-Up Parsers: Collects terminal tokens and group-reduces them back up to the root start symbol (e.g., LR family).

5. Context-Free Grammars (CFG) in Compilers

The 4-Tuple Specification
G = (V, T, P, S)
  • V: A finite set of non-terminal placeholder variables.
  • T: A finite set of terminal characters (the actual text tokens of the language).
  • P: Production rules showing how variables expand into sequences of other variables and terminals.
  • S: The single designated start symbol where S ∈ V.

6. Parse Trees vs. Abstract Syntax Trees (AST)

Structural Metric Full Parse Tree Abstract Syntax Tree (AST)
Detail Profile Highly detailed; explicitly displays every single grammar production step. Highly compact; retains only the essential structural components needed for execution.
Node Composition Internal nodes match non-terminal variables; leaves match raw terminal tokens. Internal nodes match operators (e.g., +); child nodes match operands.
Redundancy Factor Contains single-child chains and punctuation nodes (like parentheses). Eliminates formatting noise to optimize downstream compilation speed.

7. Top-Down Parsing Mechanics & LL(1)

Implementation Strategies
  • Recursive Descent: A top-down parser built using a collection of mutually recursive procedures, often requiring backtracking.
  • Predictive LL(1) Parsing: A non-recursive, table-driven parser that avoids backtracking entirely.
LL(1) Syntax Notation breakdown
  • First L → Scans the input string left-to-right.
  • Second L → Generates a Leftmost derivation tree sequence.
  • The (1) parameter → Uses exactly **one token lookahead** to pick the right production rule.
Critical MCQ Trap
  • Left Recursion Vulnerability: A top-down parser **cannot** parse a grammar that contains left-recursive rules. Doing so triggers infinite recursive loops, crashing the parser stack.

8. Bottom-Up Parsing & LR Engines

Implementation Strategies
  • Operates using **Shift-Reduce** state machines. It scans input tokens, shifting them onto a stack until the top elements match a grammar rule, at which point it reduces them back to a non-terminal variable.
  • LR Family Breakdown: L stands for left-to-right scanning; R stands for building a Rightmost derivation in reverse order.
  • LR Parsing Varieties (Ordered by increasing grammar capacity and table size):
    • LR(0) → Zero token lookahead parameters.
    • SLR(1) → Simple LR; uses structural FOLLOW sets to resolve shift-reduce conflicts.
    • LALR(1) → Lookahead LR; merges states with identical core items to save memory.
    • CLR(1) → Canonical LR; uses explicit lookahead tracking across states. **The absolute most powerful LR engine**.
Critical MCQ Comparison Trap
  • Bottom-up LR parsers are **strictly more powerful** than top-down LL parsers and can handle a significantly wider variety of grammars, including left-recursive structures.

9. FIRST & FOLLOW Calculation Sets

Definitions
  • FIRST(α): The set of all terminal symbols that can appear as the first character in a string derived from α. Can include the empty string ε.
  • FOLLOW(A): The set of all terminal symbols that can appear immediately to the right of variable A during any derivation sequence. **Can never include ε**.
Critical MCQ Traps
  • The global end-marker symbol $ must always be added to the FOLLOW set of the initial Start Symbol (S).

10 & 11. Left Recursion Elimination & Left Factoring

Left Recursion Elimination
  • Left recursion occurs when a production matches the form: A → Aα | β. This causes top-down parsers to loop infinitely.
  • To fix this, we convert the grammar into a right-recursive format by introducing a new variable:
    • A → βA'
    • A' → αA' | ε
Left Factoring
  • Used to resolve ambiguity when multiple production options share a common prefix: A → aB | aC. Without factoring, a parser doesn't know which choice to pick based on a single lookahead token.
  • We factor out the common prefix to defer the parsing decision: A → aA' and A' → B | C.

12 & 13. Syntax Directed Translation & ICG Forms

Syntax-Directed Translation (SDT)
  • Enriches a context-free grammar by attaching semantic action rules directly to production steps. This allows the compiler to evaluate expressions and verify types while building the parse tree.
Intermediate Code Generation (ICG) Formats
  • Three-Address Code (TAC): An intermediate representation where every instruction has a maximum of **three operand addresses** (one target output, two source inputs).
    • Example: t1 = a + b followed by t2 = t1 * c.
  • Quadruples: A structural array containing four explicit fields: (operator, argument_1, argument_2, result). Highly preferred for code optimization because variables can be tracked explicitly by target location.
  • Triples: Eliminates the explicit result field entirely by using array position indices to reference prior intermediate values: (operator, argument_1, argument_2). Avoids extra variable names but is harder to rearrange during optimization optimizations.

14. Code Optimization Strategies

Optimization Scopes
  • Local Optimization: Applied strictly within isolated **Basic Blocks** (linear sequences of code with a single entry point and a single exit point).
  • Global Optimization: Analyzes loops and data flow across multiple interconnected basic blocks.
Common Optimization Techniques
  • Dead Code Elimination: Removes operations or entire code blocks that can never be reached or whose outputs are never used.
  • Constant Folding: Evaluates arithmetic operations containing constant numbers at compile-time instead of generating runtime code (e.g., replacing x = 3 + 5 directly with x = 8).
  • Loop Optimization (Strength Reduction): Replaces high-overhead operations with lower-cost alternatives (e.g., swapping a multiplication operation inside a loop with an equivalent addition step).
  • Common Subexpression Elimination: Detects duplicate expressions and stores the result in a temporary variable instead of computing it twice (e.g., reusing the result of a + b).

15 & 16. Code Generation & Symbol Table Architecture

Code Generation Phase
  • The final back-end phase. Translates optimized intermediate code into target machine binary or assembly instructions. This phase depends heavily on the specific hardware's **register architectures and instruction sets**.
Symbol Table Details
  • A master database that records variable and function properties, including **data types, scope metrics, and memory layout offsets**. It must support fast insertions and lookups.
  • Implementation Standard: Built using a **Hash Table**, which provides highly efficient average-case time complexities of O(1) for operations.

17 & 18. Error Recovery Modes & Regular Expressions

Error Recovery Methods
  • Panic Mode Recovery ⭐: The most common error recovery strategy. When an error is hit, the parser drops incoming tokens one-by-one until it encounters a designated **synchronizing token** (such as a semicolon ; or closing brace }), allowing it to safely resume parsing.
  • Phrase-Level Recovery: Localized error repair. Modifies the token stream by inserting a missing character (like a semicolon) or swapping a token to let compilation continue.
Regular Expression Mapping
  • Regular expressions define token patterns during lexical analysis. For example, a standard variable identifier pattern can be written as:
    letter(letter | digit)*.

21 to 26. Core Operational Mechanics

Ambiguity Issues
  • A grammar is ambiguous if a single input string can produce **two or more distinct parse trees**. Ambiguous grammars break compiler predictability because they create multiple valid interpretation paths for the same line of code.
Operator-Precedence Parsing
  • A robust bottom-up parsing variant designed specifically for mathematical expressions. It works by establishing strict mathematical operator precedence hierarchies.
Directed Acyclic Graphs (DAG) ⭐
  • A specialized data structure used during optimization phases. It identifies duplicate operations within basic blocks, serving as a core tool for **Common Subexpression Elimination**.
Backpatching Techniques
  • An optimization technique used to generate code for control-flow statements (like loops or if-else conditions) in a single pass. It leaves target jump addresses blank and fills them in later once those destination labels are generated.

29. Parser Power Order Relations

Mathematical Capability Ranking
CLR(1) > LALR(1) > SLR(1) > LL(1)
  • CLR(1): Offers the highest grammar parsing capacity but creates large parsing tables that consume significant memory.
  • LALR(1): Merges equivalent states inside CLR tables to keep memory usage low while retaining near-identical parsing power. This makes it the standard choice for automated parser generators (like YACC or Bison).

⚠ Critical CIL MT Compiler Design MCQ Traps Checklist

  • Lexical Analyzers use **DFAs** to process text streams; **Syntax Parsers** rely on **Context-Free Grammars** running on underlying PDA models.
  • Top-down LL parsers **cannot handle left-recursive rules** without falling into infinite loops. You must eliminate left recursion before parsing.
  • Bottom-up LR parsers are **strictly more powerful** than top-down LL parsers and can handle left-recursive grammars out of the box.
  • FIRST sets collect valid starting terminal characters; FOLLOW sets collect terminal characters that can appear immediately to the right of a variable.
  • The global end-marker character $ is always added to the FOLLOW set of the initial start symbol.
  • Symbol tables are implemented using **Hash Tables** to ensure fast O(1) average-case operations.
  • Three-Address Code (TAC) serves as a clean, machine-independent Intermediate Representation (IR) format.
  • Directed Acyclic Graphs (DAGs) are optimization structures used to eliminate duplicate subexpressions.
  • Ambiguity means a grammar can produce **two or more distinct parse trees** for a single string.
  • Panic Mode Error Recovery handles errors by skipping tokens until it finds a reliable synchronizing token (like a semicolon).
  • Left Factoring rearranges a grammar to resolve choices that share a common prefix, making it easier for lookahead parsers to pick the right path.
