DBMS Technical Specification

Relational foundations, dependency closure mathematics, isolation anomalies, and disk storage architectures.

1. System Fundamentals & ER Modeling

DBMS: Centralized Data Control
File System: Redundant / Isolated Files
ER Model: Conceptual Structural Blueprint
Cardinality: Mapping Constraints Ratio
DBMS vs. File System Properties
  • File System Disadvantages: Suffer from physical data dependence, structural data redundancy, storage inconsistency, lack of atomicity for concurrent operations, and poor security isolation.
  • DBMS Advantages: Guarantees systemic logical/physical data independence through a 3-schema architecture (External, Conceptual, Internal views). Provides centralized optimization, declarative query execution, and robust access controls.
Entity-Relationship (ER) Components
  • Entity: A definable real-world object. Collected into *Entity Sets* (represented visually as rectangles). **Weak Entities** lack a primary key of their own and depend on an identifying strong entity set (represented as double rectangles).
  • Attributes: Characteristic properties of entities (represented as ellipses).
    • *Simple vs. Composite:* Single independent values vs. components that can be broken down further (e.g., Address broken into Street, City).
    • *Single-valued vs. Multi-valued:* Single values vs. arrays of multiple values (e.g., Phone Numbers; represented as double ellipses).
    • *Derived:* Values calculated dynamically from other existing data (e.g., Age computed from Date of Birth; represented as dashed ellipses).
  • Relationships: Associations linking multiple entities together (represented as diamonds).
  • Cardinality Ratios: Mapping constraints that restrict how many relationship instances an entity can participate in: **$1:1$, $1:N$, $N:1$, or $M:N$**.
Critical Design Traps:
  • ER-to-Relational Structural Reduction Rules: Binary $M:N$ relationships **always require a dedicated junction table** containing foreign keys pointing to both participating entities. Conversely, $1:N$ relationships can be optimized by avoiding a separate table and simply embedding the primary key of the "$1$" side as a foreign key column directly inside the "$N$" side table.

2. Relational Model & Relational Keys

Super Key: Uniqueness Property Only
Candidate Key: Minimal Uniqueness Set
Primary Key: Selected Non-Null Candidate
Foreign Key: Referential Integrity Variable
The Database Keys Hierarchy
  • Super Key (SK): Any set of one or more columns within a relation that guarantees unique identification for every row. If $K$ is a super key, then any superset of $K$ is also a super key.
  • Candidate Key (CK): A **minimal** super key. A super key is a candidate key if and only if removing any column from it breaks its uniqueness property.
  • Primary Key (PK): The specific candidate key chosen by the database designer to uniquely identify rows in the table. Must be strictly unique and **cannot contain null values**.
  • Alternate Key: Any candidate key that was not selected to be the primary key.
  • Foreign Key (FK): A column or group of columns that references the primary key of another table, enforcing **Referential Integrity Constraints** across relations.
Referential Integrity Violation Rules

When rows in a referenced parent table are updated or deleted, foreign key configurations determine how the system maintains data integrity:

Action Modifier ConfigurationBehavioral Policy Effect on Foreign Key Records
ON DELETE RESTRICT / NO ACTIONRejects the delete request on the parent table if matching child rows exist.
ON DELETE CASCADEDeletes the referencing child rows from the child table automatically.
ON DELETE SET NULLUpdates the referencing columns in the child rows to NULL.
Critical Key Counting Anomalies:
  • Combinatorial Key Calculations: Given a relation $R(A_1, A_2, \dots, A_n)$ where the only candidate keys are individual single attributes $A_1$ and $A_2$: The total number of Super Keys possible is calculated by finding all subsets that contain either $A_1$, $A_2$, or both, which equals: $$\text{Total Super Keys} = 2^{n-1} + 2^{n-1} - 2^{n-2} = 3 \times 2^{n-2}$$

3. Relational Algebra & Calculus Framework

Relational Algebra: Procedural Engine
Selection (σ): Row Horizontal Filter
Projection (π): Column Vertical Filter
Calculus: Declarative Predicate Model
Relational Algebra Operations Reference
  • Selection ($\sigma_p(R)$): Filters rows from a relation based on a selection predicate condition $p$. (Horizontal partitioning mechanism).
  • Projection ($\pi_A(R)$): Extracts specified columns $A$ from a relation. **Automatically removes duplicate rows** to maintain proper mathematical set properties.
  • Cartesian Product ($R \times S$): Combines every row from relation $R$ with every row from relation $S$. If $R$ has $n_1$ rows and $S$ has $n_2$ rows, the result contains $n_1 \times n_2$ rows.
  • Natural Join ($R \bowtie S$): Combines rows from two relations that have matching values in their shared, common columns.
  • Set Operations Rules: **Union ($\cup$), Intersection ($\cap$), and Set Difference ($-$)** can only be performed if the participating relations are **Union-Compatible** (meaning they have the exact same number of columns matching identical data domains in order).
Relational Calculus Definitions
  • Tuple Relational Calculus (TRC): A declarative, non-procedural language that describes a query using predicate expressions filtering tuple variables: $$\{ t \mid P(t) \}$$ Finds all tuples $t$ that satisfy the logical predicate condition $P(t)$.
  • Domain Relational Calculus (DRC): Uses variables that target individual column domains rather than entire records: $$\{\langle x_1, x_2, \dots, x_n \rangle \mid P(x_1, x_2, \dots, x_n)\}$$
Critical Formal Quirks:
  • Safe Expressions Requirement: Relational calculus queries can accidentally define infinite result sets—such as the expression $\{ t \mid \neg(t \in R) \}$, which matches all possible data points *not* in table $R$. To prevent this, queries must be restricted to **Safe Expressions**, meaning they only return values contained within the database's known data domains.

4. Structured SQL Engine, Joins & Views

DDL: CREATE / ALTER / DROP
DML: SELECT / INSERT / UPDATE
Inner Join: Strict Condition Match Only
Outer Join: Preserves Non-Matching Rows
SQL Sub-Language Classifications
  • DDL (Data Definition Language): Defines and modifies the physical database layout. Commands include `CREATE`, `ALTER`, `DROP`, and `TRUNCATE`. DDL changes are **implicitly auto-committed** to disk.
  • DML (Data Manipulation Language): Retrieves and modifies row-level data within tables. Commands include `SELECT`, `INSERT`, `UPDATE`, and `DELETE`.
  • DCL (Data Control Language): Manages user access permissions. Commands: `GRANT` and `REVOKE`.
  • TCL (Transaction Control Language): Manages transactional processing blocks. Commands: `COMMIT`, `ROLLBACK`, and `SAVEPOINT`.
The Joins Reference Matrix
Join FrameworkBehavioral Data Processing Strategy
INNER JOINReturns only rows where the join predicate condition is met on both sides.
LEFT OUTER JOINReturns all rows from the left table, plus matched rows from the right table (pads unmatched fields with NULL).
RIGHT OUTER JOINReturns all rows from the right table, plus matched rows from the left table (pads unmatched fields with NULL).
FULL OUTER JOINCombines Left and Right outer joins, preserving all rows from both tables and padding gaps with NULL.
SELF JOINJoins a table to itself by creating distinct table aliases (e.g., matching employees to managers within an employee table).
Views, Triggers, and Assertions
  • Views: Virtual tables defined by an underlying query. They do not store physical data independently unless configured as a materialized view.
  • Triggers: Event-driven procedures executed automatically by the database engine in response to specific operations like `INSERT`, `UPDATE`, or `DELETE` on a table.
  • Assertions: Declarative database-level constraints that check boolean expressions across multiple tables to ensure data integrity. (Supported conceptually in standard SQL, but rarely implemented natively by production database engines due to high performance overhead).
Critical Query Pitfalls:
  • SQL Three-Valued Logic Evaluation: SQL evaluates logical comparisons using three possible states: `TRUE`, `FALSE`, or `UNKNOWN`. Any direct comparison with a null value (e.g., `WHERE Column = NULL` or `WHERE Column != NULL`) always evaluates to `UNKNOWN` and fails to return rows. To safely find null values, you must use the explicit operators **`IS NULL`** or **`IS NOT NULL`**.

5. Functional Dependencies & Normalization Engine

Closure: Attribute Set Reachability
1NF: Atomic Column Values
2NF: No Partial Dependencies
3NF: No Transitive Dependencies
BCNF: Left Side Must Be Super Key
Functional Dependencies (FD) Theory

An FD expression $X \rightarrow Y$ states that if two rows match on their values for attribute set $X$, they must also match on their values for attribute set $Y$.
The **Attribute Closure** ($X^+$) is the complete set of all attributes whose values can be determined from $X$ by applying Armstrong's Axioms (Reflexivity, Augmentation, Transitivity).

Database Normal Forms Reference
Normal FormCore Target Condition Constraint Rules
1NFAll column values must be strictly atomic. Multi-valued attributes and nested tables are prohibited.
2NFMust be in 1NF, and **no partial dependencies are allowed**. No non-prime attribute can depend on a proper subset of any candidate key.
3NFMust be in 2NF. For every non-trivial FD $X \rightarrow Y$, **either $X$ is a super key or $Y$ is a prime attribute** (part of a candidate key). Eliminates transitive dependencies.
BCNFA stricter version of 3NF. For every non-trivial FD $X \rightarrow Y$, **$X$ must be a super key**. Eliminated anomalies caused by overlapping candidate keys.
4NFMust be in BCNF, and cannot contain non-trivial **Multi-valued Dependencies** ($X \rightarrow\rightarrow Y$).
5NFDeals with **Join Dependencies**. A relation can be broken down into smaller tables and perfectly reconstructed without introducing spurious rows.
Critical Deconstructive Trade-offs:
  • 3NF vs. BCNF Trade-off Matrix: Any relation can be decomposed into smaller 3NF tables while guaranteeing both a **Lossless Join** property and **Dependency Preservation**. However, decomposing a relation into stricter BCNF tables guarantees a lossless join but **cannot always preserve all original functional dependencies**.

6. Transaction Management & ACID Properties

Atomicity: All or Nothing Execution
Consistency: Preserves Schema Constraints
Isolation: Concurrently Hidden States
Durability: Committed States Survive Crashes
The ACID Property Framework
  • Atomicity: Guarantees that a transaction executes completely or not at all. Managed by the storage engine's transaction log components.
  • Consistency: Ensures a transaction transitions the database from one valid state to another, maintaining all schema constraints and rules.
  • Isolation: Keeps the intermediate state changes of concurrent transactions hidden from each other. Managed by the database's concurrency control components.
  • Durability: Guarantees that once a transaction commits, its data changes are permanently saved to non-volatile storage and will survive subsequent system crashes. Managed by recovery subsystem components.
Serializability Theory
  • Conflict Serializability: A concurrent schedule is conflict serializable if it is topolgically equivalent to a sequential execution of those transactions. This is verified by checking for cycles in a **Precedence Graph**.
    A conflict occurs if two operations belong to different transactions, target the exact same data item, and **at least one of the operations is a write ($W$)**.
  • View Serializability: A less restrictive definition of schedule equivalence that focuses on matching the read, write, and final write behaviors across transactions. Every conflict serializable schedule is view serializable, but some view serializable schedules are not conflict serializable (these are characterized by the presence of **blind writes**).
Critical Isolation Anomalies:
  • Dirty Read (Read Uncommitted Data): Occurs when Transaction 1 modifies a row, Transaction 2 reads that modified row, and then Transaction 1 rolls back its changes, leaving Transaction 2 with invalid data that technically never existed.
  • Non-Repeatable Read: Occurs when Transaction 1 reads a row, Transaction 2 updates or deletes that same row and commits, and Transaction 1 re-reads the row only to find its values have changed.
  • Phantom Read: Occurs when Transaction 1 runs a query to read a set of rows matching a search criteria, Transaction 2 inserts new rows that match that criteria and commits, and Transaction 1 reruns the query only to find new "phantom" rows in the results.

7. Concurrency Control & Recovery Systems

Shared (S): Concurrent Read Access Lock
Exclusive (X): Isolated Single Write Lock
Basic 2PL: Guarantees Serializability
Strict 2PL: Prevents Cascading Rollbacks
Two-Phase Locking (2PL) Protocols
  • Growing Phase: A transaction can acquire new locks but is not allowed to release any existing locks.
  • Shrinking Phase: A transaction can release locks but cannot acquire any new ones.
  • 2PL Classifications:
    • *Basic 2PL:* Guarantees conflict serializability, but can suffer from **Cascading Rollbacks** if a transaction aborts after releasing locks on data that other transactions have already read.
    • *Strict 2PL:* Requires that all **Exclusive (X) locks** be held until the transaction fully commits or aborts, preventing cascading rollbacks.
    • *Rigorous 2PL:* Stricter still, requiring **all locks (both Shared and Exclusive)** to be held until the transaction finishes.
Log-Based Recovery Mechanics

The system records all data modifications to an append-only **Write-Ahead Log (WAL)** on stable storage before modifying the actual database files on disk. During recovery after a crash, the system processes log records using two groups:

  • UNDO List: Contains transactions that were still active or uncommitted when the crash occurred; their partial changes are reversed to restore data consistency.
  • REDO List: Contains transactions that successfully committed before the crash; their changes are reapplied to ensure durability.
  • Checkpoints: Periodic operations that flush all dirty log buffers and database pages to disk. This bounds the recovery process, as the system does not need to re-evaluate log records written before the latest completed checkpoint.
Critical Locking Traps:
  • Deadlocks in 2PL: While the Two-Phase Locking protocol guarantees conflict serializability, it **does not prevent deadlocks**. Deadlocks occur when transactions fall into a cyclic wait condition, each holding a lock the other needs. Deadlocks must be handled using detection (checking Wait-For Graphs) or prevention strategies (like the Wait-Die or Wound-Wait timestamp schemes).

8. Physical Disk Storage & Indexing Architectures

B-Tree: Data Pointers in Internal Nodes
B+ Tree: Data Pointers in Leaf Nodes Only
Leaf Links: Sequential Range Scan Chain
Fan-out: High Node Branching Factor
B-Tree vs. B+ Tree Architectural Structures
  • B-Tree Structure: A self-balancing search tree where internal nodes store search keys alongside their corresponding record pointers. This can reduce lookup times for frequently accessed records near the root, but limits the node's branching factor (fan-out).
  • B+ Tree Structure: Internal nodes store search keys exclusively to route lookups; **all actual data record pointers are stored in the leaf nodes**. This architecture maximizes the node branching factor, allowing the tree to remain short and reducing disk I/O operations.
  • Leaf Pointer Chain: Leaf nodes in a B+ Tree are linked together sequentially in a doubly-linked chain. This allows for highly efficient **range query scans**, whereas a standard B-Tree requires expensive cross-level tree traversals to perform range lookups.
B+ Tree Node Structural Constraint Equations

For a B+ Tree node of order $p$ stored in a disk block of size $B$ bytes, where each search key takes $K$ bytes, each block pointer takes $P$ bytes, and each data record pointer takes $D$ bytes:

  • Internal Node Order Constraint Equation: $$p \cdot P + (p - 1) \cdot K \le B$$
  • Leaf Node Order Constraint Equation: $$q \cdot K + q \cdot D + P \le B$$
Critical Structural Limits:
  • Node Fill Constraints: Except for the root node, all internal and leaf nodes in a B/B+ tree of order $p$ **must always remain at least half-full** to maintain balance. Each internal node must contain at least $\lceil p/2 \rceil$ block pointers, failing which triggers structural underflow merges or key redistribution operations.
