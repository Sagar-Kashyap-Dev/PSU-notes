C Language Core Specification

A full analytical reference framework mapping standard compilation behaviors, evaluation models, and system behaviors.

1. Lexical Elements (Tokens, Keywords, Variables, Data Types)

Smallest Unit: Token
Keywords: 32 Standard (C89)
Variable: L-value Address Named
Data Types: Primitive / Derived / User-defined
Tokens & Keywords Theory
  • Tokens: The basic building blocks of a C program that the compiler reads during its lexical analysis phase. Tokens are categorized into **Identifiers, Keywords, Constants, String Literals, Operators, and Punctuators**.
  • Keywords: Reserved words that have predefined meanings to the compiler and cannot be repurposed as user-defined identifiers. There are **32 standard keywords** defined in C89 (e.g., `auto`, `volatile`, `register`).
  • Identifiers: User-defined names for variables, functions, and arrays. Must begin with a letter or an underscore `_`, and are case-sensitive.
Variables & Hardware Data Types
  • Variables: A named location in memory that holds a value. It represents an explicit **L-value** (a memory address where data can be written) and an **R-value** (the actual data stored at that address).
  • Data Type Sizes & Alignment Rules (Typical 32/64-bit Systems):

Critical Compilation Traps:
  • Identifier Naming Restrictions: Keywords cannot be used as identifiers. For example, `int float;` will fail to compile, but `int Float;` is perfectly valid because identifiers are case-sensitive.
  • Uninitialized Variables: Local variables declared within functions are not automatically cleared to zero. Instead, they contain arbitrary **garbage values** left over in memory, which can lead to unpredictable runtime behavior if read before being initialized.

Short-Circuit Evaluation Rules
  • Logical AND (`&&`): Evaluates from left to right. If the left operand evaluates to `0` (false), the entire expression is guaranteed to be false, so the **right operand is completely skipped** and not evaluated at all.
  • Logical OR (`||`): Evaluates from left to right. If the left operand evaluates to a non-zero value (true), the entire expression is guaranteed to be true, so the **right operand is completely skipped** and not evaluated at all.
Type Conversion Rules
  • Implicit Conversion (Arithmetic Promotion Hierarchy): The compiler automatically promotes mixed-type operands to the highest precision type in the expression to prevent data loss before performing an operation: $$\text{char} \rightarrow \text{int} \rightarrow \text{unsigned int} \rightarrow \text{long} \rightarrow \text{unsigned long} \rightarrow \text{float} \rightarrow \text{double}$$
  • Explicit Casting: A manual type conversion specified by the programmer using a cast expression (e.g., `(int)var`). This allows explicit type conversion, which can intentionally truncate or change data representations.
Critical Compilation Traps:
  • Side-Effects in Short-Circuiting: Be careful with expressions inside short-circuiting operators. For example, in `int x = 0; if (x && ++y)`, the `++y` expression is never executed, meaning variable `y` retains its original value.
  • Integer Division Truncation: Dividing two integers in C always produces an integer result, throwing away any fractional remainder. For example, `5 / 2` evaluates to `2`. To get a floating-point result like `2.5`, at least one operand must be explicitly cast or written as a float: `5.0 / 2`.

3. Control Statements & Loop Semantics

Branching: if-else / switch-case
Switch Condition: Integral Constants Only
Entry-Controlled: while / for loops
Exit-Controlled: do-while (Runs ≥ 1 Time)
Conditional Branching Mechanics
  • if-else Statements: Evaluates a conditional expression. If the expression yields a non-zero value, the `if` block executes; otherwise, the `else` block executes. Dangling `else` branches always match with the nearest un-isolated `if` block.
  • switch Statement Constraints: Directs execution path control across distinct `case` labels.
    • The control expression inside a `switch()` statement **must evaluate to an integer or character type**. Floating-point types (`float`, `double`) are strictly prohibited.
    • Case labels must be **compile-time constant expressions**. Variables cannot be used as case values.
    • Fall-Through Behavior: Once execution jumps to a matching case label, it will continue executing code down into subsequent cases unless stopped by an explicit `break;` statement.
Loop Mechanics & Interruption Directives
  • while Loop: An entry-controlled loop that evaluates its condition before executing the loop body. If the condition starts out false, the body does not execute at all.
  • for Loop Structure: Combines initialization, condition verification, and iteration steps into a single control line: `for(initialization; condition; increment/decrement)`.
  • do-while Loop: An exit-controlled loop that executes its body first before evaluating the condition. This guarantees that the loop body **will always execute at least once**.
  • break vs. continue:
    • `break;` immediately terminates the innermost loop or switch statement, jumping execution out past the block.
    • `continue;` skips the rest of the current loop iteration and jumps execution straight to the loop's condition/update step to start the next pass.
Critical Compilation Traps:
  • The Accidental Assignment Trap: Writing `if (x = 5)` instead of `if (x == 5)` uses the assignment operator instead of equality. This assigns `5` to `x`, which evaluates to `5` (true), causing the `if` block to always execute regardless of what `x` originally was.
  • Missing Switch Breaks: Forgetting a `break;` at the end of a case block causes execution to cascade into the next case's code. This unexpected fall-through behavior is a frequent source of logic bugs.

4. Functions & Recursion Execution Engine

Call Convention: Call-by-Value Only
Call-by-Reference: Simulated via Pointers
Stack Frame: Activation Record Allocations
Recursion: Base Case Requirement
Parameter Passing Conventions
  • Call-by-Value: C's native parameter passing mechanism. It copies the *value* of the argument directly into the function's local parameter variable. Modifications made to the parameter inside the function **have no effect on the original argument variable** in the calling function.
  • Call-by-Reference Simulation: C simulates pass-by-reference by passing the *memory address* of a variable (a pointer) to the function instead. This allows the function to modify the original variable's value by dereferencing the pointer.
Activation Records & Stack Architecture

Every function call triggers the allocation of an **Activation Record (Stack Frame)** on the runtime call stack. This frame stores parameters, local variables, and the return address link. When a function finishes and returns, its stack frame is popped and destroyed.

Recursion & Tail Call Properties
  • Recursion: An algorithmic pattern where a function calls itself, directly or indirectly, to break a problem down into smaller sub-problems. Every recursive function **must have a base case** to stop the recursion; omitting it causes infinite recursion and results in a **Stack Overflow** crash.
  • Tail Recursion: A specific recursive pattern where the recursive call is the absolute final action performed in the function before returning. Advanced compilers can optimize tail-recursive functions into simple loops, reusing the same stack frame to save memory.
Critical Compilation Traps:
  • Returning Local Addresses: Never return the address of a local variable from a function (e.g., `return &local_var;`). Once the function exits, its stack frame is destroyed, turning that returned pointer into a **dangling pointer** targeting invalid or reused memory.
  • Static Variables in Recursion: Local variables declared as `static` inside a recursive function are not re-allocated with each call. Instead, a single instance is shared across all recursive frames, meaning changes to a static variable persist across all levels of the recursion.

5. Arrays & String Mechanics

Array Layout: Contiguous Sequence
Array Name: Constant Pointer to Index 0
String Type: Null-Terminated char Array
Sentinel Value: '\0' (ASCII 0) Mark
Array Contiguity & Decay Properties
  • Array Foundations: A collection of identical data types stored sequentially in a continuous block of memory. Elements can be accessed instantly by index.
  • Pointer Decay Behavior: In most contexts, an array's name implicitly "decays" or evaluates to a **constant pointer targeting its first element** (`&array[0]`).
    For an array `int A[10];`, writing `A` is functionally equivalent to writing `&A[0]`.
  • Multi-Dimensional Addressing: For a 2D array defined as `int A[M][N];`, the memory location of element `A[i][j]` is computed using a row-major layout formula: $$\text{Address}(A[i][j]) = \text{Base Address} + \left(i \times N + j\right) \times \text{sizeof(int)}$$
String Architecture & Standard Library Utilities
  • String Definition: C does not have a native string data type. Strings are implemented as simple **arrays of characters terminated by a null character `\0`** (ASCII value 0).
  • The string `\"GATE\"` takes up **5 bytes** of storage because the compiler automatically appends the trailing `\0` sentinel character to mark the end of the string.
  • Standard Library String Functions (``):
    • `strlen(s)`: Counts and returns the number of characters in a string up to, but not including, the terminating `\0` character.
    • `strcpy(dest, src)`: Copies characters from the source string into the destination buffer, including the trailing `\0`.
    • `strcat(dest, src)`: Appends the source string onto the end of the destination string, overwriting the destination's original `\0` character.
    • `strcmp(s1, s2)`: Performs a lexicographical comparison of two strings character by character. Returns `0` if they match perfectly, a positive value if `s1 > s2`, or a negative value if `s1 < s2`.
Critical Compilation Traps:
  • Array Bounds Violations: C does not perform any runtime bounds checking on arrays. Accessing an index outside an array's declared size (e.g., accessing `A[12]` on a 10-element array) results in **undefined behavior**, which can corrupt data or cause segmentation faults.
  • String Literals vs. Character Arrays: Modifying a string defined as a pointer literal (like `char *ptr = "Hello"; ptr[0] = 'M';`) causes a runtime crash because literals are typically stored in read-only memory. To create a modifiable string, declare it as a standard character array instead: `char arr[] = "Hello"; arr[0] = 'M';`.

6. Advanced Pointers & Arithmetic Blueprint

Pointer: Variable Storing Address
Scale Factor: sizeof(*ptr) Bytes
Double Pointer: Pointer to a Pointer
Function Pointer: Execution Code Address
Pointer Operators & Arithmetic Rules
  • Operators: The address-of operator `&` extracts a variable's memory address, while the dereference operator `*` accesses the value stored at a pointer's target address.
  • Pointer Arithmetic Scaling Rules: Adding or subtracting an integer value $K$ to a pointer does not shift its value by exactly $K$ bytes. Instead, the address is scaled based on the size of the data type the pointer targets: $$\text{New Address} = \text{Current Address} + \left(K \times \text{sizeof}(\text{*pointer})\right)$$
  • Subtracting one pointer from another within the same array returns the **number of array elements between them**, rather than the raw number of bytes separating them.
Double Pointers (Pointers-to-Pointers)

A double pointer (e.g., `int **ptr;`) stores the memory address of another pointer variable. Double pointers are commonly used to update a pointer variable inside a function call, or to manage dynamic 2D array structures.

Function Pointers Syntax & Logic

Function pointers store the entry address of executable code sections in memory, allowing code to invoke functions dynamically.
Syntax template: `void (*fptr)(int);` declares a pointer named `fptr` that can target functions taking an integer parameter and returning void.

Critical Compilation Traps:
  • Operator Precedence Layout Mistakes: The expression `*ptr++` reads the value at `ptr` first and then increments the pointer address itself, because postfix `++` has higher precedence than unary `*`. Conversely, `(*ptr)++` increments the actual value being targeted by `ptr` without changing the pointer's address.
  • The Void Pointer Limitation: A generic void pointer (`void *`) can store any object's address, but it **cannot be dereferenced directly or used in pointer arithmetic** without first being explicitly cast to a specific type, since the compiler does not know the underlying data type's size.

7. User-Defined Structures (Structs, Unions, Enumerations)

Struct: Independent Member Locations
Union: Shared Member Memory Space
Enum: Named Integer Constant List
Padding: Hardware Alignment Insertion
Structures & Hardware Data Alignment Padding
  • Structure (`struct`): A user-defined data type that groups related variables of different types into a single named structure. Each member is allocated its own distinct memory location within the structure.
  • Data Alignment & Padding Rules: Modern CPUs read memory in aligned word chunks (e.g., 4 or 8 bytes) rather than individual bytes. To optimize access speeds, compilers insert unused bytes—called **padding**—between structure members to ensure each data type aligns with natural memory boundaries.
  • The total size of a structure can be larger than the sum of its individual members' sizes due to these padding bytes.
Unions Memory Overlay Architecture
  • Union (`union`): A user-defined data type where **all members share the exact same starting memory address**.
  • The total size allocated for a union is determined by the size of its **single largest member**. Modifying one union member overwrites the shared memory space, corrupting the data values of any other members.
Enumerations (`enum`)

Enumerations define a user-specified type whose values are restricted to a list of named integer constants. By default, the first identifier is assigned a value of `0`, and each subsequent name increments by `1` (e.g., `enum State { FALSE, TRUE };`), unless specific values are manually assigned.

Critical Compilation Traps:
  • Self-Referential Struct Declarations: A structure cannot contain an instance of itself as a direct member, as this would create an infinitely recursive size. However, a structure **can contain a pointer to an instance of itself**, which is the foundational building block for linked data structures like nodes in linked lists or trees.
  • Union Member Overwrite Risks: Remember that a union can only hold one valid member value at any given time. Reading a member after modifying a different member will interpret the raw bytes through the lens of the newly requested data type, often yielding garbage results.

8. Dynamic Memory Allocation & Storage Classes

malloc: Allocates Raw Garbage Bytes
calloc: Allocates Zero-Cleared Blocks
realloc: Resizes Extends / Relocates Blocks
free: Reclaims Heap Allocations
Dynamic Heap Allocator Core APIs (``)
  • `malloc(size_t size)`: Allocates a continuous block of memory of the specified size in bytes on the heap. It returns a generic `void *` pointer targeting the start of the block. The allocated memory is uninitialized and **contains garbage values**. If allocation fails, it returns `NULL`.
  • `calloc(size_t num, size_t size)`: Allocates memory on the heap for an array of elements. It initializes the entire block's bits to **zero**. Takes slightly longer to execute than `malloc` due to this initialization step.
  • `realloc(void *ptr, size_t new_size)`: Resizes an existing heap allocation block. It attempts to expand or contract the block in-place. If adjacent memory isn't available, it allocates an entirely new block elsewhere, copies the existing data over, automatically frees the old block, and returns the new pointer.
  • `free(void *ptr)`: Releases an allocated block of heap memory back to the system. Passing a pointer that has already been freed—or one that wasn't returned by an allocator API—causes undefined behavior and can result in **double-free memory corruption** crashes.

Critical Compilation Traps:
  • Memory Leaks: A memory leak occurs when dynamically allocated heap memory is orphaned without being freed. This happens when the last pointer tracking an allocation address is reassigned or goes out of scope, leaving the block unreachable and permanently consuming memory.
  • Dangling Pointers: Calling `free(ptr)` releases the memory block, but does not modify the pointer variable itself. The pointer continues to store the now-invalid address. Accessing or dereferencing this **dangling pointer** results in undefined behavior. It is safe practice to set pointers to `NULL` immediately after freeing them.
  • The Address-of Register Restriction: Variables declared with the `register` keyword cannot have the address-of operator `&` applied to them, because CPU registers do not reside in system RAM and lack a physical memory address.

9. Preprocessor Directives, File Handling & Command Line Arguments

Preprocessor: Text Substitution Engine
Macros: No Runtime Function Overhead
FILE: System Stream Control Block Descriptor
argv[0]: Execution Command String Link
Preprocessor Directives Pipeline
  • The preprocessor is a text-substitution engine that handles directives starting with a hash symbol `#` before compilation begins. It strips comments and modifies source text without evaluating code logic.
  • Macro Substitution (`#define`): Replaces instances of a macro identifier with a specified replacement string.
    Macros do not have function call overhead, but they do not perform parameter type checking.
  • Conditional Directives (`#ifdef`, `#ifndef`, `#endif`): Controls which sections of source code are passed to the compiler, which is useful for writing cross-platform code or preventing duplicate header files.
File Handling Streams Engine (``)
  • Files are processed through an opaque structure type called a `FILE` stream pointer descriptor.
  • Core File Operations:
    • `fopen("file.txt", "r")`: Opens a communication stream. Modes include `"r"` (read), `"w"` (write, which truncates existing file contents), and `"a"` (append). Returns `NULL` if the file cannot be opened.
    • `fclose(FILE *fp)`: Closes an open file stream, ensuring all buffered data is written to disk.
    • `fscanf() / fprintf()`: Performs formatted input and output operations on file streams.
Command Line Arguments Passing

Arguments passed to a program at execution are captured by the parameters defined in `main()`:
int main(int argc, char *argv[])

  • `argc` (Argument Count): An integer that tracks the total number of arguments passed, including the program's executable name.
  • `argv` (Argument Vector): An array of character pointers (strings) representing each argument. The first entry, **`argv[0]`**, always stores the name or path of the running executable program.
Critical Compilation Traps:
  • Macro Evaluation Ordering Bugs: Macros are expanded as raw text substitutions, which can lead to expression evaluation bugs if parameters are not wrapped in parentheses. For example, `#define SQUARE(x) x * x` translates `SQUARE(2 + 3)` into `2 + 3 * 2 + 3`, which evaluates to `11` instead of `25`. To fix this, wrap macro arguments in parentheses: `#define SQUARE(x) ((x) * (x))`.
  • Missing EOF Verifications: When reading files character by character, always check for the `EOF` (End of File) sentinel value using an integer type (`int ch`), rather than a `char` type, to ensure the sentinel value can be correctly recognized.
