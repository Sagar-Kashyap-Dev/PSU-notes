1. Software Development Life Cycle & Requirements

SRS: Software Requirements Specification
Functional: System Input/Output Behavior Rules
Non-Functional: Security, Latency, & Scaling Constraints
The Engineering Phase Sequence

The Software Development Life Cycle (SDLC) defines a clear phase progression for software construction:

$$\text{Feasibility Analysis} \longrightarrow \text{Requirements Gathering} \longrightarrow \text{System Architecture} \longrightarrow \text{Implementation} \longrightarrow \text{Testing} \longrightarrow \text{Deployment \& Maintenance}$$
Requirement Engineering Pipeline
  • Elicitation & Analysis: Interacting with stakeholders to discover real domain constraints, separating unambiguous operational functions from qualities.
  • Specification (SRS Document): The formal contract describing what the system must achieve without detailing how it will be implemented internally.
  • Validation: Reviewing the SRS alongside users to prevent design errors early, ensuring requirements are complete, consistent, and traceable.

3. Design Architecture Principle Matrices

Cohesion: Intra-module strength functional focus
Coupling: Inter-module dependency connectivity
Goal: High Cohesion & Low Coupling
Complexity: McCabe Cyclomatic Control Paths
The Cohesion Scale (Low/Worst to High/Best)
  1. Coincidental (Worst): Elements are grouped randomly within a module with no real relationship.
  2. Logical: Elements perform tasks that are logically similar (e.g., grouping all I/O handling together) but structurally distinct.
  3. Temporal: Elements are grouped because they execute at the same time during initialization or error routines.
  4. Procedural: Elements execute in a specific order of steps.
  5. Communicational: Elements execute in order and operate on the same shared data pool.
  6. Sequential: The output of one element serves as the direct input to the next ($A \to B$).
  7. Functional (Best): The module performs exactly one clearly defined, atomic task.
The Coupling Scale (High/Worst to Low/Best)
  • Content Coupling (Worst): One module directly modifies or accesses the internal data/code of another module.
  • Common Coupling: Multiple modules share global data stores.
  • Control Coupling: One module passes flag variables to explicitly direct the execution path of another module.
  • Stamp Coupling: Modules pass complex composite data structures (like an entire object or struct) when they only need a few individual fields.
  • Data Coupling: Modules communicate purely by passing simple parameters or basic data types.
  • No Coupling (Best): Modules are completely independent and share no boundaries.
McCabe Cyclomatic Complexity Mathematics

Measures the structural complexity of a program's Control Flow Graph (CFG):

$$V(G) = E - V + 2P$$ *Where $E$ represents graph edges, $V$ represents nodes, and $P$ represents connected components (typically $P=1$ for a single program).*

Alternatively, calculated via predicate nodes ($C$) or regions ($R$):

$$V(G) = C + 1 = R$$

4. Testing Lifecycles & Validation Layers

Unit: Atomic function isolated tests
Integration: Interface bridge connection tests
System: End-to-end full-stack checks
Acceptance: User validation signature tests
The Verification and Validation V-Model

Testing maps step-by-step to corresponding architectural design phases:

  • Unit Testing: Targets isolated methods or classes. Uses **Stubs** (simulated lower-level modules called by the unit) and **Drivers** (simulated higher-level calling frameworks).
  • Integration Testing: Verifies communication across module interfaces. Strategies include *Top-Down* (requires stubs), *Bottom-Up* (requires drivers), and *Big Bang* (integrating everything at once; highly discouraged).
  • System Testing: Validates the completely assembled application against global requirements.
  • Acceptance Testing: Performed by the client. Includes **Alpha Testing** (conducted at the developer's site) and **Beta Testing** (conducted by users in real-world environments).

6. Unified Modeling Language (UML) Behavioral Systems

Structural: Static Class Diagrams
Behavioral: Dynamic Activity Sequences
Actors: External Boundary Entities
Core UML Diagram Modalities
  • Use Case Diagrams: Capture system functional boundaries from an external user's perspective, mapping connections between Actors and specific Use Cases using `<>` or `<>` relationships.
  • Class Diagrams: Map the static data schema of the application, detailing class fields, methods, and structural associations (e.g., Generalization, Aggregation, and deep lifecycle Composition).
  • Activity Diagrams: Model operational control flow step-by-step, capturing concurrency using parallel execution forks and joins.
  • Sequence Diagrams: Capture dynamic object interactions over time, tracing message exchanges along vertical lifelines.
