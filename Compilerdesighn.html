1. Compiler Phases & Structural Workflow

Analysis: Front-End (Machine-Independent)
Synthesis: Back-End (Machine-Dependent)
Passes: Phased Pipeline Optimization Groups
The Complete Pipeline Architecture

A compiler processes a source program through linear translation phases. The Symbol Table and Error Handler communicate directly with every single phase:

  1. Lexical Analyzer (Scanner): Transforms raw source characters into validated tokens.
  2. Syntax Analyzer (Parser): Group tokens into hierarchical parse trees based on Context-Free Grammars.
  3. Semantic Analyzer: Performs type checking, scope tracking, and semantic rule validation.
  4. Intermediate Code Generator: Emits clean, machine-independent low-level code (e.g., Three-Address Code).
  5. Code Optimizer: Minimizes execution latency and memory footprint of the intermediate structure.
  6. Target Code Generator: Maps refined operations into machine-specific assembly or object instructions.

2. Lexical Analysis & Finite Automata Engine

Token: Syntactic Category Type Class
Lexeme: Raw Source Value String Instance
Pattern: Regular Expression Rule Matcher
Engine: Lex/Flex Automated Scanning Generators
Automata Mechanics in Scanning
  • Lexical match rules are defined using Regular Expressions and translated into execution paths via **Deterministic Finite Automata (DFA)**.
  • Lookahead Guard Buffer: Uses a trailing buffer system (e.g., evaluating `if` vs `ifa`) to track input pointers until a longest matching legal prefix token is identified.
  • Lexical Errors: Triggered when the input character stream fails to match any established regular transition path (resolved via strategies like character insertion, deletion, or substitution).

3. Top-Down Parsing & LL(1) Grammars

LL(1): Left-to-right | Leftmost | 1 Lookahead
Left Recursion: $A \to A\alpha \mid \beta \implies$ Infinite Loop Failures
Left Factoring: Resolving common prefix paths
FIRST & FOLLOW Formulations

These sets guide production choice in predictive parsing tables:

  • $\text{FIRST}(\alpha)$: The set of terminals that begin strings derived from $\alpha$. If $\alpha \xrightarrow{*} \epsilon$, then $\epsilon \in \text{FIRST}(\alpha)$.
  • $\text{FOLLOW}(A)$: The set of terminals that can appear immediately to the right of variable $A$ in some sentential form: $$\text{If } S \text{ is start symbol, } \$ \in \text{FOLLOW}(S)$$ $$\text{For production } A \to \alpha B \beta, \text{ add } (\text{FIRST}(\beta) \setminus \{\epsilon\}) \text{ to } \text{FOLLOW}(B).$$ $$\text{If } \beta \xrightarrow{*} \epsilon, \text{ add all elements of } \text{FOLLOW}(A) \text{ to } \text{FOLLOW}(B).$$
LL(1) Matrix Constraints

A grammar is LL(1) if and only if for every pair of distinct productions $A \to \alpha \mid \beta$:

  1. $\text{FIRST}(\alpha) \cap \text{FIRST}(\beta) = \emptyset$ (No overlapping starter terminals).
  2. If one production derives $\epsilon$ (e.g., $\beta \xrightarrow{*} \epsilon$), then $\text{FIRST}(\alpha) \cap \text{FOLLOW}(A) = \emptyset$.

The Complete Parser Hierarchy Power Scale

Bottom-up parsers construct a rightmost derivation in reverse by executing **Shift** and **Reduce** stack operations. Power limits scale as follows:

$$\text{LR}(0) < \text{SLR}(1) < \text{LALR}(1) < \text{CLR}(1)$$
Parser Strategy TypeTotal Operational State Space BoundsLookahead Context Resolution Target
LR(0) / SLR(1)Identical state space size (Uses $I_0$ sets)SLR(1) restricts reduction actions strictly to $\text{FOLLOW}(A)$ cells.
CLR(1)Much larger state space (Splits items based on lookahead context)Extremely powerful; handles complex language constructs but uses a massive table.
LALR(1)Identical state space size to LR(0)Merges CLR(1) states that share identical core item structures, resolving lookahead discrepancies.
Critical Table Conflict Conditions:
  • Shift/Reduce (S/R) Conflict: The engine cannot decide whether to shift the next input token or execute a production reduction.
  • Reduce/Reduce (R/R) Conflict: The engine matches multiple valid productions and cannot decide which reduction to prioritize.

5. Semantic Analysis & Syntax-Directed Translation (SDT)

Synthesized: Computed up from node children
Inherited: Computed down from parent/sibling nodes
S-Attributed: Uses synthesized attributes exclusively
L-Attributed: Inherits only from left siblings or parents
Attribute Dependency Vectors
  • S-Attributed SDT: Evaluated bottom-up alongside standard **LR parsing operations**. All semantic actions are executed immediately at the end of production reductions.
  • L-Attributed SDT: Can evaluate synthesized or inherited attributes, provided dependencies do not flow from right-to-left. L-attributed specifications can be cleanly evaluated during top-down **LL(1) parsing passes**.

6. Intermediate Code Generation (ICG) Matrix

AST: Abstract Syntax Tree
DAG: Directed Acyclic Graph Value Multiplexer
Three-Address: Max 3 Operands per Instruction
Three-Address Code (3AC) Storage Layouts

Computes operations using an explicit form: $x = y \text{ op } z$. Storage implementations include:

  • Quadruples: Uses explicit records tracking: $\langle \text{op}, \text{arg1}, \text{arg2}, \text{result} \rangle$.
  • Triples: Eliminates explicit temporary variable names by referencing operations via table row indices: $\langle \text{op}, \text{arg1}, \text{arg2} \rangle$.
  • Indirect Triples: Decouples operations from the execution sequence using an array of pointers to stable triple rows.

8. Symbol Tables, Error Recovery & YACC Systems

Symbol Table: Scope & Type Info Hash Map
Panic Mode: Discard tokens until a delimiter is hit
YACC / Bison: LALR(1) Structural Parser Generators
Error Recovery Models
  • Panic-Mode Error Recovery: The parser discards incoming tokens until it encounters a designated **synchronizing token** (such as a semicolon or closing brace). This isolates the error and allows the parser to resume syntax checking.
  • Phrase-Level Recovery: Replaces an erroneous input prefix with a local correction string (e.g., appending a missing semicolon) to allow continuous processing.
YACC Implementation Architecture

YACC (`Yet Another Compiler-Compiler`) converts a Context-Free Grammar specification into a high-performance **LALR(1) parser engine**:

  • Uses the token output stream from `Lex` scanners to execute actions. Conflict tracking defaults to resolving shift/reduce ambiguities in favor of **shifting**, and reduce/reduce ambiguities by picking the **first declared production**.
